VS Queue Monitor
Vintage Story client log queue monitor
VS Queue Monitor — Help
Live tail requirements
Live tail requires Edge/Chrome with the File System Access API. The app polls the picked log file handle and reads new bytes as it grows.
Picker blocked by “system files”
Browsers may block protected/system locations (commonly parts of %APPDATA% on Windows). Workaround: expose the log under a non-system folder (e.g. Documents/home) using a hard link to the log file, a folder junction, or a symlink — then pick the exposed path. If the picker still complains after a junction, paste the full path to the .log file in “Generate an exact command” so the app suggests a direct hard link first.
Windows (hard link to log file — try first)
if not exist "%USERPROFILE%\Documents\vs-queue-monitor" mkdir "%USERPROFILE%\Documents\vs-queue-monitor"
mklink /H "%USERPROFILE%\Documents\vs-queue-monitor\client-main-1a2b3c4d.log" "%APPDATA%\VintagestoryData\Logs\client-main.log"
One file directly under vs-queue-monitor (name includes an 8-char hash — filled in by the generator). Requires the real log file to exist and to be on the same drive as your Documents folder.
Windows (folder junction example)
if not exist "%USERPROFILE%\Documents\vs-queue-monitor" mkdir "%USERPROFILE%\Documents\vs-queue-monitor"
mklink /J "%USERPROFILE%\Documents\vs-queue-monitor\data-1a2b3c4d" "%APPDATA%\VintagestoryData"
Do not mkdir the data-… / logs-… name first — mklink /J creates it. Then pick: %USERPROFILE%\Documents\vs-queue-monitor\data-1a2b3c4d\Logs\client-main.log — the data-xxxxxxxx suffix is a hash of your pasted path (filled in by “Generate an exact command”).
Empty folder? A normal empty logs-… / data-… is usually not a junction — often mkdir created that name first (then mklink could not run), or mklink was never run (e.g. PowerShell). Delete that folder if empty (rmdir), then run only the generator lines in cmd. In cmd, dir shows real junctions with type <JUNCTION>, not <DIR> alone. If the link works but files are missing, regenerate — older builds could mis-handle paths under a subfolder of Logs.
Linux / macOS (file symlink example)
mkdir -p ~/vs-queue-monitor/logs-9f8e7d6c
ln -s ~/.config/VintagestoryData/Logs/client-main.log ~/vs-queue-monitor/logs-9f8e7d6c/client-main.log
Then pick: ~/vs-queue-monitor/logs-9f8e7d6c/client-main.log — same hash idea from the generator.
Why we can’t show the exact blocked path
For security, browsers don’t reveal the exact filesystem path you attempted to pick when access is blocked. The commands above use typical Vintage Story locations; replace the source path if yours differs.
Generate an exact command (paste your real path)
Paste the full path to your log file (Windows) or the log file path (Linux), or a data folder if you are not pasting a .log path — then copy the generated command.
On Windows, if you paste a .log path, the generator prefers a mklink /H hard link to a single file in vs-queue-monitor (e.g. client-main-xxxxxxxx.log — same drive as Documents; log file must exist). Otherwise it suggests a mklink /J junction (data-… / logs-…). Mac/Linux use symlinks under ~/vs-queue-monitor/.
